# 둘 다 · 내 PKM으로 이식

> 라이브에서 만든 v1·v2는 운영자 인프라(헤르메스 에이전트 + 사전 셋팅 저장소)에서 동작했습니다. v3는 그 리포트 루틴을 [2회차부터 굴려온 본인 PKM 저장소](/week2/setup)로 옮겨 직접 운영합니다. 習 리포트와 學 렌즈, 두 커맨드를 같은 저장소에 둡니다.

## v3의 본질 — 주도권 이전

운영자 인프라는 라이브 기간 동안만 동작합니다. 챌린지 이후에도 학습 리포트 루틴을 계속 굴리려면 **본인 통제 하에 v3가 굴러가는 상태**가 되어야 합니다.

옮길 곳은 이미 있습니다. 2회차에 셋팅한 본인 PKM 저장소(`my-brain-v1`)입니다. 본인 학습 데이터가 실제로 쌓이는 곳은 운영자 저장소가 아니라 본인 `outbox/`와 `.ai-wiki/`입니다. 리포트는 그 데이터 옆에 있어야 합니다. 데이터와 리포트가 한 저장소에 있어야 챌린지가 끝나도 같이 자랍니다.

## 무엇이 어디로 가는가

| | v1·v2 (운영자 인프라) | v3 (본인 PKM) |
|---|---|---|
| 실행 | 헤르메스 에이전트 | 본인 Claude Code의 `/리포트` 커맨드 |
| 자료 소스 | [learning-templates / learning-report](https://github.com/imakerjun/learning-templates/tree/main/learning-report) | 본인 `outbox/` + `.ai-wiki/` |
| 결과물 도착 | 배포 URL · 스크린샷 (디스코드 DM) | 본인 `.inbox/` |

2회차의 네 커맨드(`/쓰기` · `/수집` · `/돌아보기` · `/주간정리`) 옆에 다섯 번째 커맨드 `/리포트`가 들어갑니다. 결과물을 `.inbox/`에 두는 이유는 [`/주간정리`와 같습니다](/week2/setup#2-커맨드-둘러보기). AI가 만든 분석 자료는 본인이 직접 쓰는 `outbox/`와 섞지 않습니다.

## 이식 프롬프트 초안

본인 PKM 저장소를 Claude Code로 열고, 아래 초안을 그대로 던집니다. 고칠 곳은 한 곳, 3번의 대괄호입니다. [v2의 12개 분석](/week5/v2) 전부가 아니라 **본인에게 가장 도움이 됐던 한 가지**만 골라 붙여넣습니다.

```text
이 저장소는 내 PKM 저장소야. outbox/는 내가 직접 쓴 글,
.ai-wiki/는 LLM용 복제본, .inbox/는 raw 자료와 AI 산출물을 두는 곳이야.

https://github.com/imakerjun/learning-templates/tree/main/learning-report 의 리포트 구조를 참고해서,
매일 학습 리포트를 만드는 슬래시 커맨드를 이 저장소에 이식해줘.

1. .claude/commands/리포트.md 로 만들어줘.
2. 자료 소스는 outbox/ 최근 7일 글과 .ai-wiki/index.md야.
   운영자 저장소 데이터가 아니라 내 데이터만 읽어.
3. 리포트에 아래 분석을 반드시 포함해줘.

   [v2에서 가장 도움이 됐던 분석 프롬프트 1개를 여기에 붙여넣기]

4. 결과물은 .inbox/ 에 {오늘 날짜}-리포트.md 로 저장해줘.
   AI 산출물이니까 outbox/에는 쓰지 마.
5. 표는 쓰지 말고, 전체 한 페이지 이내로.
6. 커맨드를 다 만들면 지금 한 번 실행해서 첫 리포트를 보여줘.
```

에이전트가 learning-report 구조를 읽고, 본인 저장소에 맞춘 `/리포트` 커맨드를 만들어 첫 리포트까지 돌려줍니다. 응답이 빗나가면 빗나간 지점 한 줄을 짚어 다시 던집니다.

## 매일 아침 자동까지

`/리포트`는 본인이 호출하고 싶을 때 돌려도 동작합니다. v1·v2처럼 매일 아침 도착까지 가고 싶다면 한 줄 더 던집니다.

```text
/리포트 커맨드를 매일 아침 7시(한국 시간)에 자동 실행하고 싶어.
이 머신에서 가능한 방법(crontab · launchd 등)을 먼저 제안하고, 내가 고르면 설정해줘.
```

도착 경로를 `.inbox/` 밖으로 넓히고 싶다면(메일 · 디스코드 DM 등) 같은 프롬프트에 한 줄 추가하면 됩니다.

## 學 렌즈도 같은 자리로 (`/렌즈`)

習 리포트만 옮기면 절반입니다. [學 · 문서 렌즈](/week5/lens)도 같은 방식으로 `/렌즈` 커맨드가 됩니다. 공식문서 URL 하나를 주면 여러 렌즈로 변형해 학습용으로 배포하는 커맨드입니다.

```text
이 저장소는 내 PKM 저장소야.

https://github.com/imakerjun/learning-templates/tree/main/doc-lenses 의 문서 렌즈 구조를 참고해서,
공식문서 한 페이지를 여러 렌즈로 변형해 학습용으로 배포하는 슬래시 커맨드를 이식해줘.

1. .claude/commands/렌즈.md 로 만들어줘.
2. 입력은 내가 주는 URL이나 문서 한 개야.
3. 결과물(문서 렌즈)을 배포하고 공유 URL과 스크린샷을 보여줘.
4. 커맨드를 다 만들면 지금 한 번 실행해서 보여줘.
```

2회차의 네 커맨드 옆에 `/리포트`(習)와 `/렌즈`(學)가 나란히 들어가면, 두 사이클이 모두 본인 통제 하에 굴러갑니다.

## 검증

`/리포트`를 한 번 실행 → `.inbox/`에 오늘 날짜 리포트 파일이 생겼는가 → 내용이 v2에서 받던 형태와 비슷한가. 자동화까지 갔다면 다음 날 아침 정말 도착하는가.

**입력 가설**: 다음 날 아침 본인 `.inbox/`에 v3 리포트가 쌓인다 → 챌린지가 끝나도 학습 리포트 루틴이 본인 통제 하에 살아남은 상태입니다.

<Callout type="info">
**v3가 챌린지의 진짜 종착점** — v1은 운영자 인프라에서의 동작 확인, v2는 운영자 인프라에서의 실험, v3는 본인 PKM으로의 이식. 데모데이에서 발표할 **4주 누적 한 단어**는 이 단어가 챌린지 끝난 후에도 자라는가가 핵심이고, 그건 v3가 본인 데이터 옆에서 굴러가야 가능합니다.
</Callout>
