# 3. 저장소 셋팅

> 앞 페이지에서 본 `outbox/` + `ai-wiki/` 구조를, 본인 컴퓨터에 10분 안에 깝니다. **셋업 자체가 학습은 아닙니다.** 빨리 끝내고 [4번](/week2/outbox)에서 본인 데이터 만지러 가는 게 목적.

이 페이지를 따라가면 끝났을 때 본인에게 남아 있는 것은 세 가지입니다.

1. `~/git/my-brain-v1/` 폴더 (clone 완료)
2. `outbox/`에 본인 손으로 넣은 글 1개
3. `/reflect`를 한 번 돌려서 `ai-wiki/`에 복제된 결과물 1개

이 셋이 안 보이면 셋업이 안 끝난 겁니다. 다음 페이지로 넘어가지 마세요.

## 1) 저장소 clone

starter 저장소 [`imakerjun/my-brain-v1`](https://github.com/imakerjun/my-brain-v1)을 clone합니다. `inbox/` · `outbox/` · `ai-output/` 골격과 첫 outbox 템플릿이 이미 들어가 있어 손으로 만들 게 없습니다.

```bash
cd ~/git    # 또는 본인이 작업하는 폴더
git clone https://github.com/imakerjun/my-brain-v1.git
cd my-brain-v1
```

clone이 끝나면 폴더 구조가 이렇게 보입니다.

```
my-brain-v1/
├── inbox/             # raw 자료 격리 — 정리 X, 그대로 던지는 곳
│   └── README.md
├── outbox/            # 사람용 영역 — 본인이 직접 쓰는 글 (append-only)
│   └── 첫-outbox.md   # 복사해서 본인 첫 글로 바꾸는 템플릿
├── ai-output/         # AI 산출물 격리 — 라인 단위로만 outbox로 옮김
│   └── README.md
└── README.md          # 폴더 의미 + 작성 원칙
```

여기서 **본인이 직접 쓰는 자리는 `outbox/` 한 곳**입니다. `inbox/`는 raw 자료를 던져두는 임시 자리, `ai-output/`은 AI가 만든 산출물 격리. 헷갈리면 [2번 메커니즘](/week2/mechanism#1-핵심-메커니즘--한-저장소에-두-영역)을 한 번 더 보세요.

<Callout type="tip">
`outbox/`에 본인 PR·노션·미팅 기록이 들어간다면 `.gitignore`에 `outbox/`를 추가해 로컬에만 두는 걸 권장. 공개 push 여부는 본인 판단입니다.
</Callout>

### 라이브 직전 보강 2가지

mechanism의 두 영역 골격을 살리려면 `ai-wiki/`와 `.claude/commands/` 두 가지가 더 필요합니다. 라이브에서 같이 깝니다.

```diff
my-brain-v1/
  ├── inbox/
  ├── outbox/
+ ├── ai-wiki/                # LLM 친화 자동 복제본 — /reflect가 채움
  ├── ai-output/
+ ├── .claude/
+ │   ├── commands/
+ │   │   ├── reflect.md      # outbox → ai-wiki 동기화 + 메타인지 1줄
+ │   │   └── digest.md       # 7일 누적 회고
+ │   └── settings.json       # 권한 · 기본 모델
  └── README.md
```

<Callout type="warning">
라이브 전에 my-brain-v1을 미리 clone해뒀다면, 라이브 중에 `git pull`을 한 번 더 받아 위 두 가지 보강을 받아갑니다.
</Callout>

## 2) 커맨드 둘러보기

`.claude/commands/` 안에 들어 있는 슬래시 커맨드 두 개를 한 번씩 열어봅니다. 본인이 라이브 후 직접 고치게 될 자리.

| 커맨드 | v1 | 역할 | 언제 호출 |
|---|---|---|---|
| `/reflect` | ✅ | `outbox/`의 최근 글을 읽고 → `ai-wiki/`에 시간순 prefix로 복제 + 본문 3줄(메타인지 1줄 · 액션 1줄 · 근거 1줄)을 새 outbox 글로 생성 | 본인 리듬에 맞춰 (PR 한 편 끝낸 뒤 · 자기 전 · 일요일 오후. 시간 제약 X) |
| `/digest` | ✅ | 최근 7일 `outbox/` 글을 한 페이지로 누적 · 패턴 1줄 짚음 | 일요일 수동 |

라이브에서는 `/reflect`만 v1에 넣고, `/digest`는 미션 7일을 채운 다음 주에 본인이 처음 돌려봅니다.

<Callout type="info">
**커맨드 본문은 프롬프트 한 장**입니다. `.claude/commands/reflect.md` 파일을 열면 자연어 markdown이 그대로 들어 있어요. 라이브 후 본인 outbox 패턴을 보고 본문 한 줄씩 고치는 게 4회차로 이어지는 작업.
</Callout>

`/reflect`의 v0 본문은 [4번 페이지](/week2/outbox#커맨드-v0--reflect)에 그대로 박혀 있습니다. 셋업 단계에서는 본문을 읽기만 하고, 손으로 고치는 건 outbox에 본인 글이 한 편 쌓인 뒤에.

## 3) 셋업 검증 1줄

clone만 하고 다음으로 넘어가면 v1이 안 돕니다. **본인이 손으로 한 번 굴려보고 결과물이 눈에 보이는 시점까지** 셋업입니다.

<Steps>
<Step title='outbox에 글 1개 넣기'>
`outbox/첫-outbox.md`를 복사해서 `outbox/2026-05-20-셋업-테스트.md`로 이름을 바꾸고, [6원칙](/week2/mechanism#3-outbox-한-글의-형식--6원칙)의 첫 줄 주장 한 줄만 본인 것으로 바꿉니다. 예: *"나는 PKM이 아니라 PIM에 빠져 있었다. 폴더만 깔끔하고 다시 안 봤기 때문이다."*
</Step>
<Step title='/reflect 한 번 돌리기'>
Claude Code에서 `/reflect` 입력. `ai-wiki/`에 같은 글이 시간순 prefix(`2026-05-20-셋업-테스트.md`)로 복제되는지 봅니다.
</Step>
<Step title='ai-wiki에 파일 1개 떴는지 본인 눈으로 확인'>
`ls ai-wiki/` 또는 에디터에서 폴더 열기. 파일 1개가 보이면 셋업 끝. 안 보이면 권한 설정(`settings.json`) 또는 커맨드 본문에서 막힌 것. 라이브에서 같이 짚습니다.
</Step>
</Steps>

<Callout type="tip">
🎯 검증의 본질은 "동기화가 동작하는가"가 아니라, **본인이 `outbox/`에 첫 글을 한 줄 적었는가**입니다. 그 첫 줄이 오늘의 outbox 첫 데이터.
</Callout>

## 다음 주에 추가될 후보 (참고)

v1에는 안 들어갑니다. 본인이 outbox를 7일 굴려본 뒤, 막힌 지점이 보이면 그때 하나씩 깝니다.

| 이름 | 종류 | 역할 |
|---|---|---|
| `/draft <주제>` | 커맨드 | outbox에 6원칙 템플릿으로 새 글 골격만 생성 (첫 줄 주장은 빈 칸) |
| `/critique` | 커맨드 | outbox 최근 글을 AI가 **약점만 짚어줌**. "다시 써줘" X · "내가 약한 부분 짚어줘" O ([2번 메타 룰](/week2/mechanism#outbox-경계--단위는-내-손-다듬기는-ai-손)) |
| `/ingest <link>` | 커맨드 | 외부 URL · PR · 노션 한 페이지를 `inbox/`에 raw 그대로 저장 |
| `korean-spell-check` | 스킬 | 한국어 맞춤법 다듬기. 단락 통째 재작성 X, 글자 단위 교정만 |
| `humanizer` | 스킬 | AI 톤으로 빠진 문장을 본인 톤으로 되돌림 (메이커준 본인 글로벌 스킬) |

<Callout type="info">
스킬은 Claude Code 전역(`~/.claude/skills/`)에 깔리는 자산입니다. 학습 에이전트 한 저장소만의 것이 아니라 본인이 평소에 쓰는 도구 세트의 일부 — 한 번 깔면 다른 프로젝트에서도 같이 작동합니다.
</Callout>
