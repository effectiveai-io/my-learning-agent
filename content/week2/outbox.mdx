# 4. outbox 설계 + 데이터 + 커맨드

> outbox가 척추입니다. **본인의 생각 1줄**을 먼저 작성하고, 그걸 구현하기 위한 처리와 `.inbox/`를 거꾸로 설계해 나갑니다.

## 핵심 원칙 3가지

### 1. `.inbox/`는 정리하지 않는다

원자료를 그대로 던져넣습니다. 분류 X, 태깅 X, 폴더 깊이 X. **정리는 처리 단계에서 LLM이 한다.** 폴더가 점(`.`)으로 시작하는 이유도 같음 — 사람이 정리할 자리가 아니라 커맨드가 알아서 다룰 자리.

### 2. `outbox/`에서 거꾸로 설계한다

"본인이 받고 싶은 1줄"이 `outbox/`다. `outbox/`가 정해지면 처리가 따라오고, 처리가 정해지면 `.inbox/` 구조가 따라온다.

<Callout type="tip">
대부분의 PKM이 망하는 이유는 **inbox부터 설계하기 때문**. 노트 폴더 구조 짜다가 3개월이 지나간다.
</Callout>

### 3. 처리는 LLM이 잘 읽을 수 있어야 한다

- 평문 마크다운
- 시간순 (파일명 prefix)
- 링크는 절대 경로 또는 명시적 ID
- 메타데이터는 frontmatter 최소

## `.inbox/` + `.ai-wiki/` — 카파시 LLM wiki 패턴

이 두 폴더의 출처는 [안드레 카파시의 LLM wiki](/week2/appendix). "질의 때마다 검색하는 RAG" 대신, **LLM이 마크다운 위키를 직접 만들고 매번 갱신**하는 패턴입니다. 사람은 소스를 던지고 질문하고, LLM은 페이지를 쓰고 상호 참조를 유지합니다.

### 3 레이어

| 레이어 | 우리 폴더 | 누가 쓰나 |
|---|---|---|
| **raw 소스** | `.inbox/` | 사람. PR·노션·노트·AI digest를 던져넣기만. 수정 X. `/수집` 커맨드가 한 줄로 던지는 자리. |
| **위키** | `.ai-wiki/` | LLM. `/수집`(type: raw)과 `/돌아보기`(type: reflected)가 호출마다 새로 채움. 사람은 직접 안 만짐. |
| **스키마** | `CLAUDE.md` + `.claude/commands/{쓰기·수집·돌아보기·주간정리}.md` | 사람+LLM. 컨벤션·워크플로우 정의. |

### 3 동작 (개념)

카파시가 분류한 **위키의 세 동작**입니다. 우리 구현은 4개 커맨드가 분담합니다.

| 동작 | 무엇인가 | 우리 구현 |
|---|---|---|
| **Ingest** | 새 소스를 위키에 녹이기 | `/수집 <URL\|텍스트>` 한 줄 → `.inbox/`에 raw 저장 + `.ai-wiki/`에 `type: raw`로 미러. 사람-소화 글이 들어가는 다른 자리(`outbox/`)는 `/쓰기`가 6원칙 골격을 깔고 본인이 본문을 채우면 `/돌아보기`가 `type: reflected`로 동기화. |
| **Query** | 위키에 질문하기 | `/돌아보기`가 `.ai-wiki/`의 두 type을 컨텍스트로 읽고 다음 outbox 골격 한 편(주장·액션·근거 3줄)을 제안. 본인은 받아 적지 않고 한 줄씩 검토. |
| **Lint** | 위키 건강 검진 | 아직 X. `.ai-wiki/` 고립 페이지·모순·낡은 주장 점검은 다음 주 후보 (`/약점짚기` 등). |

<Callout type="tip">
사람이 위키를 포기하는 이유는 **유지보수 부담**. LLM은 지치지 않고, 상호 참조를 잊지 않고, 한 번에 15개 파일을 건드린다. 유지보수 비용이 0에 가까워지니 위키가 살아남는다.
</Callout>
