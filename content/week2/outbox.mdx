# 4. outbox 설계 + 데이터 + 커맨드

> outbox가 척추입니다. **"매일 아침 받고 싶은 1줄"**을 먼저 정하고, 그걸 만들기 위한 처리와 inbox를 거꾸로 설계합니다.

<Callout type="warning">
이 페이지는 **본문 채워야 함** — 커맨드 프롬프트 v1과 라이브 데모 스크립트 보강 예정.
</Callout>

## 핵심 원칙 3가지

### 1. inbox는 정리하지 않는다

원자료를 그대로 던져넣습니다. 분류 X, 태깅 X, 폴더 깊이 X. **정리는 처리 단계에서 LLM이 한다.**

### 2. outbox에서 거꾸로 설계한다

"매일 아침 받고 싶은 1줄"이 outbox다. outbox가 정해지면 처리가 따라오고, 처리가 정해지면 inbox 구조가 따라온다.

<Callout type="tip">
대부분의 PKM이 망하는 이유는 **inbox부터 설계하기 때문**. 노트 폴더 구조 짜다가 3개월이 지나간다.
</Callout>

### 3. 처리는 LLM이 잘 읽을 수 있어야 한다

- 평문 마크다운
- 시간순 (파일명 prefix)
- 링크는 절대 경로 또는 명시적 ID
- 메타데이터는 frontmatter 최소

## 라이브 25분 흐름

<Steps>
<Step title='데이터 소스 1개 픽 · 5분'>
PR 5개 · 노션 3페이지 · 노트 1폴더 중 가장 손에 잡히는 것 하나.
</Step>
<Step title='inbox/에 넣기 · 5분'>
복붙 · 다운로드 · 링크 — 어떤 방식이든. 정리 X, 그냥 넣기.
</Step>
<Step title='커맨드 1개 돌리기 · 10분'>
`/morning` 또는 `/digest`. outbox/에 1줄 메타인지 + 1줄 액션이 나오는지 확인.
</Step>
<Step title='맞다/아니다 즉시 판정 · 5분'>
"이게 진짜 내가 모르고 있던 거 맞나?" 본인이 즉시 짚기. 틀리면 프롬프트 한 번 더.
</Step>
</Steps>

## 커맨드 v0 — `/morning`

```markdown
# /morning

inbox/ 의 최근 7일 데이터를 읽고 outbox/{오늘 날짜}-morning.md 파일을 생성한다.

## 출력 형식

다음 3줄로만 작성한다. 길게 쓰지 않는다.

1. **오늘 나는 뭘 모르고 있나?** — 한 줄
2. **그래서 오늘 뭐 할까?** — 10분 안에 시작할 수 있는 행동 1줄
3. **근거 인용** — inbox/ 어느 파일의 어느 줄에서 나왔는지 명시

## 작성 원칙

- 추측이 아니라 inbox/ 데이터에서 찾은 증거 기반
- "잘 모르겠어요" 금지. 데이터가 부족하면 "더 모을 데이터" 1줄 제안
- 매일 같은 패턴 X. 어제 outbox/와 다른 각도면 그것도 한 줄 메모
```
